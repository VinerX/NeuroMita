# RAG и Графовая память — Руководство пользователя

## Что такое RAG?

**RAG (Retrieval-Augmented Generation)** — система, которая перед каждым ответом LLM автоматически ищет релевантную информацию из прошлых разговоров и воспоминаний персонажа, и подставляет её в контекст. Это позволяет модели «вспоминать» факты, которые давно вышли за пределы окна контекста.

**Простая аналогия:** Представьте, что перед каждым ответом помощник заглядывает в записную книжку и перечитывает нужные заметки. RAG — это такая автоматическая записная книжка.

---

## Как это работает (шаг за шагом)

### 1. Вы пишете сообщение

Когда вы отправляете сообщение персонажу, система сначала запускает RAG-поиск по вашему тексту.

### 2. Поиск релевантной информации

Система ищет сразу несколькими способами (параллельно):

| Метод | Что делает | Аналогия |
|-------|-----------|----------|
| **Векторный поиск** | Ищет по смыслу (семантическое сходство) | «Найди записи, похожие по теме» |
| **Полнотекстовый (FTS)** | Ищет по точным словам (BM25) | «Найди записи, где есть эти слова» |
| **Ключевые слова** | Сопоставляет ключевые слова | Резервный метод для записей без эмбеддингов |
| **Граф знаний** | Ищет связанные сущности | «Кто с кем связан и как» |

### 3. Оценка и ранжирование

Найденные результаты оцениваются по нескольким критериям:

- **Семантическое сходство** (K1 = 1.0) — насколько контекст похож по смыслу
- **Ключевые слова** (K5 = 0.6) — совпадение конкретных слов
- **Приоритет** (K3 = 0.5) — приоритет воспоминания (critical/high/normal/low)
- **Участники** (K4 = 0.5) — совпадение говорящих/целей в диалоге
- **Давность** (K2 = 0.3) — новые записи чуть важнее старых
- **Лексический** (K6 = 0.3) — BM25-скор полнотекстового поиска

Итоговый скор = K1×sim + K2×time + K3×prio + K4×entity + K5×kw + K6×lex + шум

### 4. Вставка в промпт

Лучшие результаты (по умолчанию до 8) вставляются в системный промпт перед отправкой модели:

```xml
<relevant_memories>
- [0.89] (memory, prio=High, date=2025-03-20) Алиса очень любит кошек
- [0.76] (memory, prio=Normal, date=2025-03-19) Обсуждали книги с Борисом
</relevant_memories>

<past_context>
- [0.85] (2025-03-21 14:30) (Алиса→Борис) Ты видел тот фильм?
- [0.71] (2025-03-21 14:15) (Борис) Да, мне очень понравился
</past_context>
```

### 5. LLM генерирует ответ

Модель видит эти блоки и может использовать информацию из них для более осмысленного ответа.

## Графовая память

### Что это?

Графовая память — это **граф знаний**, который автоматически извлекает **сущности** (людей, места, предметы, концепции) и **связи** между ними из каждого диалога.

### Как это работает?

```
Диалог                    Извлечение (LLM)           Граф знаний
┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐
│ Player: Алиса   │      │ Сущности:       │      │ (Алиса)         │
│ подарила Борису │  →   │  - Алиса (person)│  →   │    │ подарила   │
│ книгу о космосе │      │  - Борис (person)│      │    ▼            │
│                  │      │  - космос (topic)│      │ (Борис)         │
│                  │      │ Связи:          │      │    │ тема        │
│                  │      │  Алиса→подарила→ │      │    ▼            │
│                  │      │    Борис         │      │ (космос)        │
└─────────────────┘      │  книга→тема→     │      └─────────────────┘
                          │    космос        │
                          └─────────────────┘
```

**После каждого сообщения** (асинхронно, в фоне):
1. Берётся текст диалога (сообщение игрока + ответ персонажа)
2. Отправляется LLM-провайдеру с промптом для извлечения сущностей
3. LLM возвращает JSON с сущностями и связями
4. Результат сохраняется в БД (таблицы `graph_entities` и `graph_relations`)

### Как граф используется при поиске?

Когда включена опция «Искать в графе знаний при RAG»:
1. Из вашего запроса извлекаются ключевые слова
2. Ключевые слова сопоставляются с именами сущностей в графе
3. Для найденных сущностей загружаются связи (1 шаг)
4. Связи форматируются как тройки: `Алиса --подарила--> Борис`
5. Добавляются в RAG-результаты наравне с воспоминаниями и историей

## Настройки

### Основные настройки RAG

| Настройка | По умолчанию | Описание |
|-----------|-------------|----------|
| **RAG включён** | Выкл | Главный переключатель. Если выключен, модель эмбеддингов не загружается |
| **Макс. результатов** | 8 | Сколько лучших результатов вставлять в промпт |
| **Порог сходства** | 0.3 | Минимальный скор для попадания в результаты |

### Настройки графовой памяти

| Настройка | По умолчанию | Описание |
|-----------|-------------|----------|
| **Экстракция сущностей** | Выкл | Включить автоматическое извлечение сущностей из диалога |
| **Провайдер для графа** | Текущий | Какой LLM-пресет использовать для извлечения (рекомендуется лёгкая модель) |
| **Искать в графе** | Выкл | Включать ли результаты из графа в RAG-поиск |

### Веса ранжирования

Веса определяют, какие факторы важнее при ранжировании результатов. Все веса настраиваются в интерфейсе:

- **Similarity (1.0)** — семантическая близость. Главный фактор
- **Keywords (0.6)** — совпадение ключевых слов
- **Priority (0.5)** — приоритет воспоминания
- **Entity (0.5)** — совпадение участников диалога
- **Time (0.3)** — свежесть записи
- **Lexical (0.3)** — полнотекстовый скор

### Режимы комбинирования

| Режим | Описание |
|-------|----------|
| **Union** (по умолчанию) | Объединение всех результатов с дедупликацией |
| **Vector only** | Только семантический поиск (быстрый) |
| **Intersect** | Только результаты, найденные ≥N методами (высокая точность) |
| **Two-stage** | Вектор для полноты, другие методы добавляют признаки |

## Рекомендации

### Быстрый старт
1. Включите RAG в настройках
2. Подождите ~1 минуту пока модель эмбеддингов загрузится
3. Поиграйте несколько сообщений — RAG начнёт подтягивать контекст

### Для графовой памяти
1. Включите «Экстракция сущностей»
2. Рекомендуется выбрать **отдельный лёгкий пресет** (например, GPT-4o-mini или Gemini Flash), чтобы не нагружать основную модель
3. Включите «Искать в графе знаний при RAG»
4. Граф заполнится автоматически по мере общения

### Оптимизация
- Если результаты слишком шумные — поднимите порог сходства (0.4–0.5)
- Если контекст слишком длинный — уменьшите макс. результатов (4–6)
- Для коротких диалогов режим **weighted** для эмбеддингов запроса работает лучше, чем **concat**
- Если нужна высокая точность — попробуйте режим **intersect**

## Модель эмбеддингов

RAG использует модель **Snowflake Arctic Embed M v2.0** для создания векторных представлений текста. Модель:
- Работает **локально** (не требует API)
- Запускается на **CPU** (не требует GPU)
- Скачивается автоматически при первом запуске
- Размер: ~110 МБ

Эмбеддинги создаются для каждого сообщения в истории и каждого воспоминания, затем хранятся в SQLite базе данных.

## Кастомизация шаблонов

Шаблоны форматирования RAG-результатов можно переопределить для каждого набора промптов. Положите файлы в папку `Structural/` набора:

- `rag_memory_item.txt` — шаблон строки для воспоминания
- `rag_history_item.txt` — шаблон строки для истории
- `rag_wrapper.txt` — обёртка для всего RAG-блока
- `graph_extraction_prompt.txt` — промпт для извлечения сущностей

### Доступные переменные в шаблонах:

**Воспоминание:** `{score}`, `{type}`, `{priority}`, `{date}`, `{content}`

**История:** `{score}`, `{date}`, `{meta}`, `{content}`, `{speaker}`, `{target}`, `{role}`

**Обёртка:** `{memory_block}`, `{history_block}`, `{graph_block}`

**Извлечение сущностей:** `{text}` — текст диалога для анализа
